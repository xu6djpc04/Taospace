## 0. 為什麼先寫方法論

先把「合理的分層」立穩，再處理具體文件（排課、畢班、培訓班…）。
這是「可複用框架」與「一次性整理」的差別。
方法論穩定後，任何新活動或新學年的文件，都照同一套規矩落地，不會走樣。

核心問題只有一句：

> **「這份東西是原始資料、查詢參考、還是整理產出？」**

答出來，就知道放哪。

## 1. 結構分層 ✅

專案由四個資料層構成，每層只管一件事：

| 層 | 路徑 | 是什麼 | 何時動它 |
|----|------|--------|---------|
| 規則 | `AGENTS.md` | 跨工具必讀規範與紅線（單一真相來源）；Claude Code 另有 `CLAUDE.md` 補充，並以 `@AGENTS.md` 自動載入本檔（人維護） | 要改助手行為或底線 |
| 輸入 | `Taofiles/` | 原始道場檔案（不修改，只讀取） | 新增原始文件時 |
| 參考 | `references/` | 外部查詢彙整、比對用的搜尋結果、方法論 | 查詢或比對時 |
| 產出 | `output/` | 整理分析後的成果（md、csv） | 每次任務完成後 |

**工具層（輔助，不屬資料生命週期）**：`scripts/` 存放可重用的 Python／PowerShell 腳本（下載、比對、轉檔、Typeless 修復等），與 `Taofiles`／`references`／`output` 三個資料層分開——它放「做事的工具」，不是「被整理的資料」。新腳本一律放 `scripts/`，不散落根目錄；一次性／已淘汰腳本移 `scripts/_archive/` 留底。

### 鐵則

- `Taofiles/` 唯讀：不刪、不改、不覆蓋原始檔。
- 產出不混回輸入；整理稿永遠另存新檔於 `output/`。
- 若原始檔有疑問，在輸出稿標注 `（待確認）`，交人決定後再修正。

## 3. 核心原則：依生命週期分層 ✅

**這是整套方法論的心臟。** 任何東西進來前，先問三句話決定它放哪：

> **「它會變嗎？該共享嗎？它是規則、參考、還是產出？」**

| 類型 | 會不會變 | 共享／版控 | 放哪 |
|------|---------|-----------|------|
| 規則／偏好 | 幾乎不變 | 是 | `AGENTS.md`（Claude Code 專屬補充見 `CLAUDE.md`） |
| 外部查詢與參考文件 | 查詢結果持續累積；方法論偶爾微調 | 是 | `references/` |
| 原始輸入 | 定版後不變 | 否（內部文件） | `Taofiles/` |
| 產出資料 | 會持續更新 | 否（歷史紀錄） | `output/` |

**鐵則**：產出資料絕不混進規則或參考（整理完的菜不放回食譜裡）。

## 4. 治理原則 ✅

讓整理工作「做得準又不出錯」，且交給任何人用都可信賴：

- **透明，不靜默**：發現資料衝突或缺漏，主動說明，不偷偷略過或自行補值。
- **不捏造**：人名、日期、職稱直接轉錄；不確定的標 `（待確認）`，不推測填補。
- **紅線明寫**：絕不可違反的事，條列在 `AGENTS.md`。
- **不外洩敏感資料**：個人聯絡方式、身分資料在公開用途的輸出檔中須確認是否保留。
- **有後果的動作先確認**：批次產出或大幅改寫前，先說明範圍，等人確認。
- **確認問句一旦出現，當輪即停**：確認問句與執行動作不得出現在同一個 response 中；問句送出後，須等待人的明確回覆才能繼續。

---

## 5. 擴充規約 🚧

新增任務類型或資料夾時的一致做法：

- **一任務一動詞**（整理／對照／摘要／彙整）；不做萬用大任務。
- 每類任務至少定義：
  1. 用途（一句話）
  2. 何時用／何時不用
  3. 輸入來源（哪個 `Taofiles/` 子資料夾）
  4. 輸出位置與命名格式
  5. 紅線（這個任務不能做的事）
- 新學年度資料夾加入 `Taofiles/` 前，確認命名符合慣例；加入後同步更新本文件第 2 節的清單。

---

## 6. 互動模式 🚧

把整理工作從「一問一答」帶到「結構化使用」：

- 呼叫固定任務類型時，先確認範圍（哪個學年、哪個活動），再執行。
- 先反問、對齊，再動手——不是丟一句就吐結果。
- 任務流程 = 把熟手腦中的 SOP，變成可重複執行的步驟。
